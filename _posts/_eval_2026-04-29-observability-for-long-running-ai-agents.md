## Motivation
Long-running agents fail in ways that traditional monitoring never catches. There's no crash, no 500-series response, and no alert firing — just an agent drifting quietly in the wrong direction over hours while every health check returns green. A misconfigured loop or a runaway retry chain can execute for an entire weekend before anyone notices, and by then the damage, whether that's wasted compute, corrupted state, or a staggering API bill, is already done.

That last scenario was the direct catalyst for this work. A surprise billing spike on a Monday morning made it clear that the observability patterns we rely on for short-lived web requests simply don't translate to agents with multi-step, multi-hour lifetimes. A web request either succeeds or fails in milliseconds; an agent can be deeply wrong for a very long time before any conventional signal surfaces. Building meaningful observability for these systems requires rethinking what we measure and when.

## Latency Targets
Measuring agent latency end-to-end is the first mistake, and it's a costly one. A full-run p95 looks like a useful signal, but it buries the real problem: you can't tell whether planning is crawling, a tool call is ballooning, or synthesis is the culprit. Set p50, p95, and p99 targets for each phase — planning, tool calls, and synthesis — independently. That granularity transforms "the agent was slow today" from a shrug into a specific, fixable problem. Latency creep is a subtler problem that phase-level targets help expose. Small per-step slowdowns that would be invisible in aggregate accumulate badly over a run with dozens of steps, so alerts should fire when a single step exceeds its phase budget rather than waiting for the full run to time out. Tracking p50 versus p99 budget per turn also makes it easier to distinguish typical behavior from tail-latency outliers that would otherwise get washed out in end-to-end averages. Separating end-to-end latency from per-tool-call latency is equally important. Slow tools have a way of hiding behind aggregate numbers, and without the breakdown you end up tuning the wrong thing. Finally, every agent that runs in production should have a hard ceiling per turn that activates a kill switch when a step goes unacceptably long. That single safeguard has saved more than a few runaway runs from turning into incidents.

## Token Accounting
Token usage is the most reliable early-warning signal for runaway agent behavior, but only if you're tracking it at the right granularity. Logging total tokens per run tells you what happened; logging prompt and completion tokens per step tells you why. The distinction matters because token-heavy steps tend to cluster around a small number of culprits — context stuffing, oversized tool responses, and unconstrained memory retrieval — and you can't address what you can't locate. Plotting cumulative token usage against timestamps reveals growth curves — a well-behaved agent's consumption should be predictable, and an upward bend is worth investigating before the run ends. Token budgets per phase are a practical guard against unbounded context expansion, catching problems before they surface as billing anomalies. Two metrics that are easy to overlook but consistently high-signal are per-turn input tokens and tool-result tokens. Per-turn input tokens show you exactly how the context window is growing across steps, which is the most direct proxy for whether your context management strategy is working. Tool-result tokens, meanwhile, often dominate total usage and are frequently underestimated during capacity planning. Keeping an eye on cache-hit rate rounds out the picture by showing how much of your token spend is actually reaching cached prefixes versus regenerating prompt content from scratch.

## Cost Modeling
Token counts are interesting; dollar costs are what get executive attention. Mapping token usage to real-time dollar cost per provider and model tier closes the loop between telemetry and business impact. The mapping itself is straightforward. The value comes from attributing costs to logical task segments rather than treating a run as a single line item. When you can say "sub-task X consumed 40% of the run's budget," you have a target for optimization rather than just a number to explain.

Cost ceilings with hard stops or human-in-the-loop gates are a practical necessity in production. Per-run and per-day limits prevent a single misbehaving run from causing outsized financial damage. They're also cheap to implement relative to the risk they eliminate. Pairing those guardrails with a cost forecast at run start gives operators an early signal when a run is tracking above its expected envelope.

We learned this the hard way. On a Saturday night, a scheduled reconciliation agent entered a retry loop after hitting a malformed record in a third-party data feed. The agent was designed to re-verify each prior step before attempting a fix. Every retry replayed the full context window from the top of the run. By Sunday morning, a job that typically cost $0.40 had consumed just under $90.

Our per-day ceiling was scoped to individual runs, not to cumulative spend across concurrent instances. Eight parallel copies of the same job each stayed just under the per-run limit. Together, they burned through the weekend budget before anyone saw the first alert. The incident cost roughly $700 before an on-call engineer killed the fleet. No meaningful work had been completed.

The post-mortem changed how we thought about measurement. Raw spend had looked manageable at the per-run level. It was only when we divided total cost by tasks actually completed that the number became alarming. Cost per completed task became the metric we built our sampling strategy around.

Workflows where that ratio was stable earned a lighter sampling rate: we'd inspect one in every twenty runs and rely on aggregate dashboards for the rest. Workflows where it was volatile, with high variance run-to-run or a steady upward drift, got promoted to a heavier regime where we traced every run in detail. The reconciliation agent stayed on heavy sampling for thirty days after the fix, until three consecutive weeks of stable readings let us dial it back.

Unlike raw spend, cost per completed task normalizes against actual output delivered. That makes it useful for comparing runs of different scope, tracking efficiency gains over time, and assessing whether a workflow is economically viable at scale. If that number is climbing while task volume holds steady, something upstream has changed, whether it's prompt length, retry policy, model selection, or context-replay logic, and you want to know before the next weekend rolls around.

## Dashboards
A good observability dashboard for long-running agents doesn't try to show everything — it surfaces the signals that matter while a run is still in progress. The core of any such dashboard is a live run timeline showing the current step, elapsed time, and token burn rate. That combination answers the question operators actually ask in the moment: where is this run, how long has it been going, and is it consuming resources at a normal pace? Alongside the timeline, a cost gauge that displays projected final cost next to actual spend so far gives operators an early window into whether a run is on track financially. A latency heatmap broken down by step type surfaces recurring slow points without requiring anyone to dig through raw logs. Rounding out the view, a drift indicator that flags when behavior is deviating from a baseline run profile catches the subtle wrongness — the agent that isn't crashing, just quietly going sideways — that none of the numeric metrics alone would catch. A latency-percentile panel — showing p50, p95, and p99 step durations side by side — makes tail latency visible at a glance, which is critical for long runs where a single outlier tool call can quietly dominate wall-clock time. A per-tool token breakdown table complements this by attributing token spend to each tool invocation, so operators can immediately see which tool is driving context growth rather than hunting through aggregated totals.

## Takeaways
Silent drift and billing surprises feel inevitable until you instrument at the step level — then they become solved problems. The patterns described here are not especially complex, but they do require a deliberate shift away from the request-scoped thinking that most observability tooling is built around. Once latency, token, and cost signals are captured together at step granularity, you have a complete health picture of a running agent rather than a post-mortem artifact.

The practical advice is to start simple. A basic dashboard with rough thresholds will surface the majority of problems, and you can refine those thresholds as you accumulate real run data. The more important point is to start at all. Observability is not an optional layer to add once an agent system matures — it is the difference between a production system you can trust and one you're perpetually guessing about.
