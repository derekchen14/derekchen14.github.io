## Motivation
The problem with long-running AI agents is that they can drift in silence for hours before anyone notices. Model behavior shifts, tool outputs accumulate unexpected side effects, and conversational state grows in ways that no single-turn evaluation ever catches. By the time something looks wrong, it usually already is.

The inciting incident for this observability work was a billing surprise at the end of a weekend agent run — a charge large enough to make it clear that gut-feel monitoring was not going to cut it. Without instrumentation baked in from the start, failure modes stay invisible right up until they become expensive or, worse, catastrophic.

The monitoring patterns that work well for traditional web services do not translate cleanly to agentic systems. An HTTP endpoint either responds or it does not; an agent session can be technically alive while consuming resources at an unsustainable rate, producing subtly wrong outputs, or spinning in a loop that looks like progress. Real-time visibility into what an agent is actually doing, at each step, is a distinct requirement — and it deserves to be treated as one.

## Latency Targets
Agentic latency is a compounding problem, not a single-call one. A p99 stall at one step multiplies across every subsequent step: a tail that looks tolerable in isolation can stretch a thirty-minute task into a three-hour one. That's why per-step SLOs at p50, p95, and p99 matter more here than they do for one-shot inference, where a slow call simply ends and you move on. Set those percentile targets alongside a total wall-clock budget for the full task, and track both. The per-step numbers catch a widening tail before it shows up in end-to-end time. Equally important is understanding where latency lives. An end-to-end breakdown often obscures the real bottleneck, so it is worth separating model latency from tool-call latency at the per-step level. A slow tool response and a slow LLM round-trip require completely different remediation paths, and conflating them delays diagnosis. On top of the alerting thresholds, every agentic system should have a hard latency ceiling that triggers a kill switch. Runaway loops are a real failure mode, and the only safe default when an agent exceeds its time budget is to abort gracefully rather than let it consume unbounded time and resources.

## Token Accounting
Token usage is the atomic unit of cost in any LLM-powered system, so tracking it at the right level of granularity is the foundation of everything else. That means recording tokens consumed per step, per tool call, and per agent session — not just a rollup at the end of a job. Within each turn, it is worth separating prompt tokens from completion tokens, since they are priced differently and tell different stories. Prompt token counts reveal how much of the context window each turn is actually consuming, which is the leading indicator of runaway context growth. When conversation history bloats the prompt unexpectedly, per-turn input tokens are usually the first metric to show it. Tool-result tokens deserve special attention — a tool response can return kilobytes of structured data that silently inflates the context on every subsequent turn. Track cache-hit rate alongside raw counts to quantify avoidable spend: a high hit rate means prompt prefixes are served from cache rather than re-encoded. Finally, logging token counts next to agent actions lets you correlate verbosity spikes with specific calls, so you can fix root causes instead of just observing symptoms.

## Cost Modeling
Token counts only become actionable when mapped to dollar spend. The translation is simple once you have a live feed of usage paired with current pricing tables. Building the right metrics on top of that feed is the harder work.

We learned that the hard way on a Saturday night. A research-summarisation agent ran a weekly batch job across roughly four hundred documents, each requiring several tool calls and a synthesis step. An upstream data issue caused those calls to return empty results. The agent read them as inconclusive rather than erroneous and entered a retry loop, re-querying the same documents with progressively longer prompts. By Sunday morning it had consumed twelve times the expected token budget. Every individual call had succeeded and every response had arrived on time, so nothing in our dashboards flagged it. The incident was invisible to every monitor we had except the invoice.

The most useful metric to emerge from that postmortem was cost per completed task. Token volume tells you how much the model is working. Cost per task tells you whether that work is producing proportionate value. In the summarisation agent's case, the per-task cost would have spiked the moment the first batch started cycling through retries without advancing the completion count. An alert set at two times the rolling average would have fired within the first hour.

Adopting cost per task also forced a harder conversation about sampling. Sixty percent of our tool calls were low-stakes lookups that did not need full-length completions. We introduced a tiered policy: concise-mode completions for lookups and re-ranking, full-length completions only for synthesis and final output. That single change cut average task cost by around forty percent with no measurable drop in quality.

Attribution is the next layer. Tagging costs by task type, user, or project makes budgeting tractable and surfaces which workloads are pulling disproportionate spend before the monthly bill arrives. Paired with rolling cost projections, teams can intervene during a long job rather than after it finishes.

The final piece is enforcement. Hard cost caps with graceful abort logic let the system shut down in a controlled way when a session crosses a threshold, preserving completed work rather than failing abruptly. Had that logic existed on the Saturday-night job, the agent would have halted after the first anomalous hour and woken an on-call engineer. Build it early.

## Dashboards
A well-designed observability dashboard for agentic systems operates at two levels: a fleet-wide overview and a per-session drilldown. At the top level, the panels that matter most are active agent count, step throughput, token burn rate, and error rate. These four metrics, tracked together, give a real-time read on whether the system is healthy without requiring anyone to dig into individual sessions. When something does look wrong, per-session drilldowns provide the detail needed to diagnose it. A full trace of each session — capturing every step, every tool called, and the token count at each turn — makes it possible to reconstruct exactly what an agent did and where things went sideways. Anomaly overlays complement this by flagging sessions that deviate from baseline cost or latency profiles, so unusual behavior surfaces proactively rather than waiting for a user to report a problem. For tooling, the options range from general-purpose infrastructure platforms like Grafana with custom OpenTelemetry pipelines to purpose-built agent observability tools like LangSmith. The right choice depends on how much of the instrumentation your team wants to own versus how much you want provided out of the box, but the underlying metrics are the same regardless of where they end up displayed. Latency-percentile views — surfacing p50, p90, and p99 step durations side by side — are especially valuable because mean latency can mask the long-tail outliers that most degrade user experience in interactive agent workflows. Per-tool token panels break down token consumption by each individual tool call, making it immediately visible which integrations are the biggest context inflators and where prompt-compression or caching efforts will have the most impact.

## Takeaways
Observability for long-running agents is not an optional enhancement to add once the system is stable. Silent drift and surprise costs are the default outcome when instrumentation is absent, and they tend to compound before they become visible. The question is not whether to invest in observability, but when — and the answer is from the start.

The practical place to begin is with token accounting and latency budgets. Both are tractable to instrument early, they generate immediate signal, and they provide the foundation that makes dashboards and cost caps meaningful rather than cosmetic. Adding those layers on top of solid per-step metrics is a natural progression, not a separate project.

Perhaps the most important framing shift is to stop treating agent sessions as single requests. Each session is a distributed trace: a sequence of steps, tool calls, and model interactions that need to be observed as a whole to be understood. Adopting that mental model early shapes instrumentation decisions in ways that pay off as systems grow in complexity.

If you have built observability into your own agent workflows, it would be worth sharing which metrics have proven most valuable in practice — the specifics vary a lot by use case, and there is still plenty to learn across the community.
