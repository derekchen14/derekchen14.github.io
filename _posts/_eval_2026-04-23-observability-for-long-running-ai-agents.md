## Motivation
Agents that run for hours can drift silently — tool loops, prompt injections, and context bloat accumulate without obvious errors, making problems nearly invisible until it's too late. The work behind this post was kicked off by a jarring billing surprise: a weekend agent run had burned through hundreds of dollars with no visibility into where the tokens went. Without observability, debugging is purely post-mortem; with it, you catch runaway behaviour before costs spiral out of control.
## Latency Targets
Before you can track latency, you need a sharp definition of "acceptable" — and for most agent workloads, p99 tail latency is that definition, not averages, because your slowest runs are the ones users actually feel. From there, establish per-step latency budgets alongside end-to-end SLOs so you have concrete thresholds to measure against rather than vague intuitions about slowness. Instrumenting turn-level timing is what makes slow tool calls or model responses surface immediately, giving you the signal you need to act before a single lagging step poisons an entire run. Critically, use percentile tracking — p50, p95, and p99 — rather than averages; tail latency is where the real pain hides, and averages will mask it every time.
## Token Accounting
Effective token accounting means tracking prompt tokens, completion tokens, and totals at both the per-turn and per-full-run level, giving you a granular picture of consumption across the agent's lifetime. Attribute those tokens to their source — system prompt, memory retrieval, tool output, or user messages — so you can pinpoint what's driving usage up. Equally important is attributing those tokens to their source — system prompt, memory retrieval, tool output injection, or user messages — so you know exactly which component is driving usage up. Set per-run token budgets and wire up alerts to fire when thresholds are crossed mid-run, turning a reactive billing review into a proactive safeguard. Finally, log token counts alongside task metadata so you can later correlate token spend with actual outcomes and determine whether the cost was justified.
## Cost Modeling
Token counts only matter when you convert them to dollar figures in real time, using current provider pricing tables to close the gap between technical metrics and business impact. Tracking cost-per-task and cost-per-outcome lets you reason about ROI concretely: not just whether the agent succeeded, but whether running it was worth the spend.

That distinction became urgent during what we now call the weekend-billing incident. A long-running research agent fired on a scheduled job every four hours across a Saturday and Sunday. It was configured to retry aggressively on ambiguous tool responses, so each run inflated its token footprint by two to three times the weekday baseline. With no one watching the dashboard, the anomaly ran undetected for 36 hours. By Monday morning, the weekend had consumed roughly 40% of the month's entire token budget.

By every functional measure, the agent had performed fine: tasks completed, no errors raised, results delivered on time. The cost-per-completed-task metric told a different story, spiking from a typical $0.11 to over $0.31 per task. That single number, invisible until we built it, forced a rethink of our sampling strategy.

Before the incident, we sampled every tenth run for detailed tracing — a reasonable cadence when all runs looked similar. Afterward, we switched to cost-adaptive sampling: any run whose projected cost exceeded 1.5× the rolling 7-day median was automatically promoted to full trace capture, regardless of whether it had produced an error. Expensive, anomalous runs got recorded in detail; cheap, routine runs stayed at the lighter rate. The result was significantly lower trace-storage overhead with no loss of coverage on the runs that mattered.

The cost-per-completed-task lens then surfaced two optimization candidates we had missed: a document-summarization tool call that accounted for 28% of total spend despite appearing in only 15% of tasks, and a prompt preamble being re-injected in full on every retry instead of being cached. Fixing both brought average task cost back below $0.12 and made the monthly spend forecast reliable enough for leadership to plan capacity against. Projecting forward from observed per-run costs and usage frequency is now a standing item in our monthly infrastructure review — a discipline the weekend incident made non-negotiable.
## Dashboards
A well-designed real-time agent run dashboard surfaces the metrics that matter most during a live run: active runs, current step, token burn rate, and elapsed time — all at a glance, without digging through logs. Pair that with historical views covering run durations, cost trends, error rates, and goal completion rates so teams can spot patterns and regressions across time. Anomaly detection should be built in: runs that exceed 2× the median cost or duration ought to auto-highlight, making outliers impossible to miss. Keep dashboards role-aware too — ops needs cost and latency front and centre, while product cares most about task completion rates and output quality. - Latency-percentile views (p50, p95, p99) expose tail-latency outliers that averages hide, making it easy to catch the slowest agent runs before they impact users.
- Per-tool token panels break down token consumption by each tool call so you can see at a glance which tools are the biggest cost drivers.
## Takeaways
Observability for agents is not optional — silent drift and surprise costs are the default without it, and the longer you wait to instrument, the more expensive the lessons get. Start with token accounting and latency budgets; dashboards follow naturally once the underlying data is flowing cleanly.

The key mindset shift is treating observability as a first-class feature built into the agent architecture from day one, not something bolted on after things go wrong. Round that out by sharing runbooks for common failure patterns so the whole team can diagnose agent issues quickly, rather than relying on one person who happened to debug it before.
